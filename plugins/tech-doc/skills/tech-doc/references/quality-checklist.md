# 기술문서 검증 체크리스트

Phase 4(검증)에서 사용한다. 각 항목에 PASS/WARN/FAIL 판정.

## 1. 팩트 검증 (FAIL 시 즉시 수정)

- [ ] 문서의 모든 서비스/API가 코드에 실제 존재하는가? (Grep 확인)
- [ ] URL, 포트, 환경변수명이 코드와 정확히 일치하는가?
- [ ] 다이어그램 화살표(호출 관계)가 실제 호출 체인과 일치하는가?
- [ ] DB명, 컬렉션/테이블명이 Entity/Model과 일치하는가?
- [ ] "자동 파이프라인"으로 기술한 것이 실제로 자동인가? (독립 API가 아닌가?)

## 2. 완전성 검증 (WARN)

- [ ] 처음 보는 사람이 "이 시스템이 뭘 하는지" 1분 내 이해 가능한가?
- [ ] 핵심 프로세스가 시작~끝까지 빠진 단계 없는가?
- [ ] 에러/예외 상황 최소 1개 언급되어 있는가?
- [ ] 각 도메인의 비즈니스 목적이 첫 줄에 있는가?

## 3. 다이어그램 검증

- [ ] 모든 핵심 프로세스에 다이어그램이 있는가?
- [ ] 다이어그램당 노드 15개 이하인가?
- [ ] Mermaid에 config(look/theme) 설정이 있는가?
- [ ] 색상 규칙(be=녹, ext=적, db=회)을 따르는가?

## 4. 안티패턴 검증 (WARN)

- [ ] Controller 클래스명이 노출되어 있지 않은가?
- [ ] Vault 경로 등 구현 디테일이 과다하지 않은가?
- [ ] "~로 추정된다" 같은 불확실 표현이 없는가?
- [ ] 이모지가 다이어그램/노드에 사용되지 않았는가?
- [ ] 같은 정보가 여러 문서에 중복되어 있지 않은가?

## 5. 가독성 검증

- [ ] 한 섹션이 3스크롤 이상 넘어가지 않는가?
- [ ] 나열형 정보가 표로 정리되어 있는가?
